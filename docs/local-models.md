# Подключение локальных моделей (LM Studio / Ollama)

Утилита умеет работать с **любым OpenAI-совместимым API**, а значит и с
локальными моделями. Плюсы локалки: бесплатно, без лимитов провайдера, данные
не уходят наружу. Минусы: нужна более-менее мощная видеокарта, ответы медленнее
облака.

Ниже — как подключить локальную модель для генерации сопроводительных писем,
фильтрации вакансий и (опционально) распознавания капчи.

---

## Как это устроено

Утилита читает **три независимых секции** конфига — по одной на задачу, так что
можно смешивать провайдеров (например, письма локально, фильтр в облаке):

| Секция конфига          | Задача                                  |
| ----------------------- | --------------------------------------- |
| `openai_cover_letter`   | Сопроводительные письма (и решение тестов) |
| `openai_vacancy_filter` | AI-фильтр вакансий (`--ai-filter`)      |
| `openai_captcha`        | Распознавание капчи (нужна vision-модель) |

У каждой секции одинаковый набор параметров:

| Параметр                | Описание                                                                 |
| ----------------------- | ------------------------------------------------------------------------ |
| `api_key`               | Ключ провайдера. Для LM Studio/Ollama ключ не нужен — впишите любое непустое значение, например `"local"`. Поле **обязано** быть непустым. |
| `base_url`              | **Полный** URL эндпоинта, обязательно с `/chat/completions` на конце.     |
| `model`                 | Идентификатор модели (должен совпадать с тем, что отдаёт `GET /v1/models`). |
| `temperature`           | 0.0 — детерминированно, 0.4–0.7 — разнообразнее.                          |
| `max_completion_tokens` | Максимум токенов в ответе.                                                |
| `rate_limit`            | Запросов в минуту (0 — без ограничения; для локалки ставьте `0`).         |

> ⚠️ **`base_url` — это полный эндпоинт.** Частая ошибка — указать
> `http://.../v1` без `/chat/completions`. Утилита шлёт POST ровно на `base_url`,
> поэтому правильный вид: `http://.../v1/chat/completions`.

---

## Сетевой нюанс для Docker

Утилита в контейнере запускает LM Studio/Ollama **на хосте**. Изнутри контейнера
`localhost` указывает на сам контейнер, а не на хост. Поэтому:

- В `base_url` используйте **`host.docker.internal`** вместо `localhost`
  (на Docker Desktop для Windows/Mac работает из коробки).
- Сервер модели должен слушать **все интерфейсы** (`0.0.0.0`), а не только
  `127.0.0.1`, иначе контейнер до него не достучится.

Если утилита установлена **не** в Docker, а локально (`pipx`/venv), то используйте
обычный `localhost` и специально настраивать bind не нужно.

## Вариант A: LM Studio

1. Установите [LM Studio](https://lmstudio.ai) и скачайте **instruct**-модель
   (см. раздел «Какие модели брать»).
2. Вкладка **Developer** (Local Server) → загрузите модель → **Start Server**.
   Порт по умолчанию `1234`.
3. Включите **«Serve on Local Network»** — это bind на `0.0.0.0`, обязательно для
   Docker.
4. Узнать точное имя модели: `GET http://localhost:1234/v1/models` (или посмотреть
   в интерфейсе). Имя в конфиге должно совпадать с ним, например `qwen2.5-14b-instruct`.

Эндпоинт для конфига:
- Docker: `http://host.docker.internal:1234/v1/chat/completions`
- Локально: `http://localhost:1234/v1/chat/completions`

---

## Вариант B: Ollama

1. Установите [Ollama](https://ollama.com).
2. Разрешите слушать все интерфейсы (для Docker):
   ```sh
   # Windows (PowerShell), затем перезапустить Ollama
   setx OLLAMA_HOST "0.0.0.0:11434"
   # Linux/Mac: export OLLAMA_HOST=0.0.0.0:11434
   ```
3. Скачайте модель:
   ```sh
   ollama pull qwen2.5:7b-instruct
   ```
4. Имя модели в конфиге = тег Ollama, например `qwen2.5:7b-instruct`.

Эндпоинт для конфига:
- Docker: `http://host.docker.internal:11434/v1/chat/completions`
- Локально: `http://localhost:11434/v1/chat/completions`

## Какие модели брать

Берите **instruct**-модели. Хорошо заходят `Qwen2.5-Instruct` (отличный русский),
`Llama-3.1-8B-Instruct`, `Gemma-2-9B`. По размеру ориентируйтесь на VRAM (квант Q4):

| Модель (Q4) | ~VRAM  | Заметка                                            |
| ----------- | ------ | -------------------------------------------------- |
| 7–8B        | ~5–6 ГБ | Быстро, хватает для фильтра и писем.               |
| 14B         | ~9–10 ГБ | Письма покрасивее, влезает в 12 ГБ.               |
| 32B         | ~20 ГБ | Только с большой картой, иначе уходит в CPU и тормозит. |

> ❗ **Не берите reasoning-модели** (`Qwen3`, `QwQ`, `*-Thinking`, `DeepSeek-R1` и
> т.п.) для этой утилиты. Они тратят весь бюджет токенов на внутренние
> «размышления» и часто **не доходят до ответа** — вы получите пустую строку.
> Особенно это ломает фильтр (`max_completion_tokens` там маленький) и решение
> тестов. Если очень хочется reasoning — придётся сильно поднимать
> `max_completion_tokens` (3000–5000) и мириться с медленной работой.

**Капча** требует **vision**-модели (`qwen2.5-vl`, `llava`, `llama3.2-vision`),
но распознавание искажённого текста у мелких vision-моделей слабое — не ждите чуда.

## Настройка через веб-панель (проще всего)

1. Запустите панель и откройте `http://localhost:8090`.
2. В сайдбаре — **🤖 AI-провайдеры**.
3. Для нужной секции заполните `base_url`, `model`, `api_key` (`local`), при
   желании `temperature`/`max_completion_tokens`/`rate_limit` (для локалки `0`).
4. Нажмите **🧪 Отправить пробный** — панель дёрнет ваш эндпоинт с текущими
   значениями (ещё до сохранения) и покажет ответ и время. Так удобно убедиться,
   что модель отвечает.
5. **💾 Сохранить все секции** — запишет в `config.json`.

---

## Настройка через config.json

Пример: письма и фильтр — локально через LM Studio (в Docker).

```json
{
  "openai_cover_letter": {
    "api_key": "local",
    "base_url": "http://host.docker.internal:1234/v1/chat/completions",
    "model": "qwen2.5-14b-instruct",
    "temperature": 0.4,
    "max_completion_tokens": 800,
    "rate_limit": 0
  },
  "openai_vacancy_filter": {
    "api_key": "local",
    "base_url": "http://host.docker.internal:1234/v1/chat/completions",
    "model": "qwen2.5-7b-instruct",
    "temperature": 0.0,
    "max_completion_tokens": 200,
    "rate_limit": 0
  }
}
```

Гибрид (фильтр локально, письма в облаке Groq) — просто укажите в секциях разные
`base_url`/`api_key`/`model`.

## Проверка

- **Из панели:** кнопка «🧪 Отправить пробный» (см. выше).
- **Из CLI (dry-run, ничего не отправляется):**
  ```sh
  hh-applicant-tool -vv apply-vacancies --ai --ai-filter light \
    --search "Python" --per-page 3 --total-pages 1 --dry-run
  ```
  В логе с `-vv` появятся строки `AI запрос:` и ответы модели.

## Производительность и память

- Держите в памяти **только одну** модель — если LM Studio/Ollama загрузят
  несколько сразу, на 12 ГБ они начнут вытеснять друг друга (своп) и всё встанет.
- **Context Length**: для фильтра `light` хватает и 4k, но для `heavy` (полный
  текст резюме + вакансии) ставьте контекст побольше (8k–16k), иначе промпт
  обрежется и ответ будет пустым.
- Скорость: 7–8B на средней карте ~30–50 ток/с, 14B ~20–30 ток/с.

## Траблшутинг

| Симптом                                             | Причина и решение                                                                                     |
| --------------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| `Connection refused` / `Network error`              | Сервер модели не запущен, или слушает только `127.0.0.1`. Включите bind на `0.0.0.0` и `host.docker.internal`. |
| `404 Not Found` от эндпоинта                        | В `base_url` забыли `/chat/completions`. Допишите.                                                     |
| Ответ **пустой** (`""`)                             | Взяли reasoning-модель, либо мало `max_completion_tokens`, либо мал Context Length. Возьмите instruct-модель / поднимите лимиты. |
| `Read timed out` / очень долго                      | Модель отвечает дольше стандартного таймаута (~15 с). Возьмите модель поменьше/побыстрее или уменьшите `max_completion_tokens`. |
| `API-ключ не задан`                                 | Поле `api_key` пустое. Впишите любое непустое значение (`local`).                                      |
| Фильтр «пропускает всё»                             | Фильтр не смог разобрать ответ и вернул «подходит». Проверьте, что модель отдаёт чистый JSON `{"suitable": true/false}` (instruct, не reasoning). |
| Капча не распознаётся                               | Нужна **vision**-модель; мелкие модели плохо читают искажённый текст.                                  |
